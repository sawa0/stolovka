# План функционала обновления из Git

## Общая концепция
Система автоматического обновления приложения из Git-репозитория с выбором между стабильной (master) и экспериментальной (AI_Master) версиями через веб-интерфейс.

## Архитектура

### 1. Backend (Python)

#### 1.1 Новый модуль `git_updater.py`
**Функции:**
- `get_current_branch()` - получение текущей ветки из Git
- `check_updates()` - проверка наличия обновлений в ТЕКУЩЕЙ ветке
- `get_current_version()` - получение текущей версии (commit hash, дата, сообщение)
- `get_remote_version()` - получение версии из удаленной ТЕКУЩЕЙ ветки
- `pull_updates()` - скачивание обновлений из ТЕКУЩЕЙ ветки
- `switch_branch(branch)` - переключение между master и AI_Master
- `restart_application()` - перезапуск приложения после обновления
- `get_commit_log(from_commit, to_commit)` - получение списка коммитов между версиями

**Логика работы:**
- Текущая ветка определяется из Git (`git branch --show-current`)
- Проверка обновлений всегда для текущей ветки
- Переключение ветки = `git checkout <branch>` + `git pull origin <branch>`

**Безопасность:**
- Проверка целостности репозитория перед обновлением
- Проверка конфликтов и локальных изменений (если есть - предупреждение)
- Валидация успешности обновления

#### 1.2 Расширение `stolovka.py`
**Новые endpoints:**
- `GET /api/update/status` - текущая ветка, версия и статус обновлений
- `GET /api/update/check` - проверка доступности обновлений для ТЕКУЩЕЙ ветки
- `POST /api/update/execute` - запуск обновления ТЕКУЩЕЙ ветки
- `POST /api/update/switch` - переключение между ветками (параметр: branch - "master" или "AI_Master")
- `GET /api/update/changelog/:from/:to` - список изменений между коммитами

**WebSocket события:**
- `update_progress` - прогресс обновления (%)
- `update_status` - статус обновления (checking, downloading, switching_branch, installing, restarting, completed, error)
- `update_log` - логи процесса обновления в реальном времени

#### 1.3 Конфигурация `config.py`
**Новые параметры:**
```python
GIT_CONFIG = {
    'repo_url': 'https://github.com/sawa0/stolovka',
    'stable_branch': 'master',
    'experimental_branch': 'AI_Master',
    'auto_check_updates': True,
    'check_interval_hours': 24
}
```

- `UpdateProgress` - индикатор прогресса обновления
- `VersionInfo` - информация о текущей ветке и версии
- `ChangelogViewer` - просмотр изменений между версиями

**Функции:**
- `getCurrentStatus()` - получение текущей ветки и версии
- `checkForUpdates()` - проверка обновлений для ТЕКУЩЕЙ ветки
- `startUpdate()` - запуск обновления ТЕКУЩЕЙ ветки
- `switchBranch(branch)` - переключение на другую ветку

### 3. Хранение настроек

**Файл `settings.json` (или аналогичный конфиг):**
```json
{
  "update_branch": "master",  // или "AI_Master"
  "auto_check_updates": true,
  "check_interval_hours": 24,
  "last_update_check": "2026-05-19T10:30:00Z"
}
```

**Примечание:** База данных НЕ используется для этой фичи. Все настройки в файле конфигурации. История обновлений доступна через `git log`.

### 4. Процесс обновления (пошагово)

#### Сценарий 1: Обновление текущей ветки

**Шаг 1: Проверка обновлений**
1. Пользователь нажимает "Проверить обновления"
2. Frontend отправляет запрос `GET /api/update/check`
3. Backend выполняет `git fetch origin` и `git branch --show-current`
4. Сравнивает локальный и удаленный commit hash ТЕКУЩЕЙ ветки
5. Возвращает информацию о доступных обновлениях

**Шаг 2: Подготовка к обновлению**
1. Проверка наличия локальных изменений (`git status`)
2. Если есть изменения - предупреждение пользователю
3. Показ changelog (список коммитов между версиями)
4. Подтверждение пользователем

**Шаг 3: Выполнение обновления**
1. Пользователь подтверждает обновление
2. Frontend отправляет `POST /api/update/execute`
3. Backend через WebSocket отправляет статусы:
   - "Проверка репозитория..."
   - "Скачивание обновлений..."
   - "Применение изменений..."
   - "Проверка целостности..."
4. Выполнение `git pull origin <current_branch>`
5. Проверка успешности операции

**Шаг 4: Перезапуск приложения**
1. Отправка уведомления "Перезапуск приложения..."
2. Закрытие всех соединений
3. Перезапуск Python процесса
4. Автоматическое переподключение frontend

**Шаг 5: Проверка после обновления**
1. Frontend переподключается через WebSocket
2. Запрашивает новую версию
3. Отображает уведомление об успешном обновлении

#### Сценарий 2: Переключение между ветками

**Шаг 1: Выбор ветки**
1. Пользователь выбирает стабильную или экспериментальную версию
2. Frontend отправляет `POST /api/update/switch` с параметром branch

**Шаг 2: Переключение**
1. Backend проверяет локальные изменения
2. Выполняет `git checkout <branch>`
3. Выполняет `git pull origin <branch>`
4. Через WebSocket отправляет прогресс

**Шаг 3: Перезапуск**
1. Перезапуск приложения
2. Frontend переподключается
3. Отображает новую текущую ветку

### 5. Обработка ошибок

#### 5.1 Возможные ошибки
- Нет подключения к интернету
- Конфликты при git pull
- Локальные изменения блокируют обновление
- Ошибка при перезапуске приложения
- Поврежденный репозиторий

#### 5.2 Стратегии восстановления
- Детальное сообщение об ошибке пользователю
- Сохранение логов всех операций
- Инструкции по ручному исправлению
- При конфликтах - предложение сделать `git stash` или откатить локальные изменения

### 6. Безопасность

#### 6.1 Права доступа
- Только администраторы могут запускать обновления
- Логирование всех попыток обновления
- Подтверждение через пароль для критических операций

#### 6.2 Валидация
- Проверка подлинности источника (origin URL)
- Проверка цифровой подписи коммитов (опционально)
- Sandbox-тестирование перед применением (опционально)

## Риски и ограничения

1. **Перезапуск приложения** - требует механизма автоматического перезапуска (systemd, supervisor, или скрипт)
2. **Конфликты файлов** - локальные изменения могут блокировать обновление
3. **Зависимости** - новая версия может требовать новые Python пакеты (нужен requirements.txt)
4. **База данных** - миграции схемы БД при обновлении нужно делать вручную или через отдельный механизм
5. **Downtime** - кратковременная недоступность при перезапуске (~5-10 секунд)

## Альтернативные подходы

1. **Docker-based updates** - обновление через pull нового Docker образа
2. **Blue-Green deployment** - запуск новой версии параллельно со старой
3. **Hot reload** - обновление без перезапуска (ограниченно для Python)
